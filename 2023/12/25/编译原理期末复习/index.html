自顶向下：

  • 最左推导：被替换的总是当前符号串中的最左非终结符。
  • 最右推导：被替换的总是当前符号串中的最右非终结符。（规范推导）

自底向上：（规约是推导的逆过程）

  • 最右规约：被规约的总是当前符号串最右。
  • 最左规约：被规约的总是当前符号串最左。(规范规约)

语法树

  • 子树：任一结点及其全部后继。

    • 直接子树（树高为1）：子树根只有直接后继。
    • 一颗语法树 <—> 多种推导序列。
    • 一颗语法树 <—> 唯一最左推导和最右推导。

  • 短语：每颗子树的叶子。

    • 直接短语：每颗直接子树的叶子。
    • 句柄：某句型的最左直接短语（规范分析中最先被规约的子串）。
    • 素短语：至少包含一个终结符且不包含更小素短语的短语。

二义性

一个文法可能有多颗语法树能够生成同一个给定的终结符号串，那么称这样的文法具有二义性。换句话来说，如果文法L(G)的某个字符串不是只有一个最左/最右推导，那么就具有二义性。

提取左公因子

对于每个非终结符号A，找出它的两个或多个选项之间的最长公共前缀α。若α≠ε，即存在一个非平凡公共前缀，那么将所有A产生式:
A -> αβ_1 | αβ_2 | … | αβ_n|r
替换为
A -> αA’ | r，A’ -> β_1 | β_2 | … |β_n
其中r表示所有不以α开头的产生式体，A’是新非终结符。

消除左递归

套用如下公式：
（原）A -> Aa | B
（消除左递归）A -> BA’ ，A’ -> aA’ | ε

FIRST集和FOLLOW集

一般书上终结符用小写，非终结符用大写。

FIRST集的求法:

First集合最终是对产生式右部的字符串而言的，但其关键是求出非终结符的First集合，由于终结符的First集合就是它自己，所以求出非终结符的First集合后，就可很直观地得到每个字符串的First集合。

  • 直接收取：对形如U－> a … 的产生式（其中a是终结符或者是ε），把a(或者ε)收入到First(U)中。
  • 间接收取：对形如U - > P 的产生式，把First(P)收入到First(U)中。
  • 反复传送：对形如U－> Y_1Y_2 … Y_k 的产生式
    • 若Y_1是非终结符，则First(Y_1)-{ε}∈First(U)。
    • 若Y_1,Y_2 … Y_i (1<=i<=k-1)均为非终结符且都能推出ε，则First(Y_1)，First(Y_2) … First(Y_{i+1})中一切非ε的∈First(U)。
    • 若Y_1,Y_2 … Y_k都能推导出ε，则ε∈First(U)。

FOLLOW集的求法:

Follow集合是针对非终结符而言的，Follow(U)所表达的是句型中非终结符U所有可能的后随终结符号的集合，特别地，“$”是识别符号的后随符。注意Follow集合是从开始符号S开始推导。

  • 直接收取：如果U是开始符号，那么$∈Follow(U)。
  • 直接收取：形如S - > … Ua … 的产生式，把a直接收入到Follow(U)中，因为a是紧跟在U后的终结符。
  • 直接收取：形如S - > … UP 的产生式，则First(P)-{ε}∈Follow(U)，且如果ε∈First(P)则Follow(S)∈Follow(U)。
  • !!反复传送：对形如S－>…aU的产生式（其中U是非终结符），应把Follow(S)中的全部内容传送到Follow(U)中。

语法制导的翻译

所谓语法制导的翻译，即使用上下文无关文法CFG来引导对语言的翻译，是一种面向文法的翻译技术。而语法制导定义SDD就是上下文无关文法的推广，因此通俗来说，本章就是将语法制导翻译成计算机可以处理的东西，方便后续的类型检查和中间代码生成。

语法制导定义

语法制导定义(Syntax-Directed Definition,SDD)是一个上下文无关文法和属性及规则的结合。属性和文法符号相关联，规则和产生式相关联。如果X是一个文法符号，a是X的一个属性，则用X.a表示属性a在某个标号为X的分析树结点上的值。

继承属性和综合属性

非终结符号有两种属性：

  • 综合属性。在分析树结点N上的非终结符号A的综合属性只能通过N的子结点或N本身的属性值来定义。 如图，E的val属性值是由其两个子结点来定义的，因此，val是E的一个综合属性。
  • 继承属性。在分析树结点N上的非终结符号A的继承属性只能通过N的父结点、N的兄弟结点或N本身的属性值来定义。 如图，L的inh是其兄弟结点T的type属性定义的，因此inh是一个继承属性。

终结符可以具有综合属性，但不能有继承属性。终结符的属性值是由词法分析器提供的词法值，在SDD中没有计算终结符号的属性值的语义规则。

属性文法

一个没有副作用的SDD有时也被称为属性文法，一个属性文法的规则仅仅通过其他属性值和常量值来定义一个属性值。而所谓的副作用，比如允许打印桌上计算器计算得到的结果，或者和一个符号表进行交互。

注释语法分析树

非终结符A有一个产生式A->XYZ，那么在语法分析树中就可能有一个标号为A的内部结点，该结点有三个子结点，从左向右的标号分别为X，Y，Z。 正式地讲，给定一个上下文无关文法，该文法的一棵语法分析树是具有以下性质的树：

  • 根节点的标号为文法的开始符号。
  • 每个叶子结点的标号为一个终结符号或ε。
  • 每个内部结点的标号为一个非终结符号。
  • 如果非终结符号A是某个内部结点的标号，并且它的子结点的标号从左至右分别为X1，X2，…，Xn，那么必然存在产生式A->X1，X2，…，Xn，其中X1，X2，…，Xn既可以是终结符号，也可以是非终结符号。

如果语法分析树各结点标记了相应的属性，则这个语法分析树就是注释语法分析树。

SDD求值顺序

依赖图

依赖图是用来确定一棵给定的语法分析树中各个属性实例的求值顺序的，注释语法分析树显示了各个属性的值，而依赖图可以帮助我们确定如何计算这些值。分析树中每个标号为X的结点的每个属性a都对应着依赖图中的一个结点；如果属性X.a的值依赖于属性Y.b的值，则依赖图中有一条从Y.b的结点指向X.a的结点的有向边。

由于F.val=digit.lexval，因此digit.lexval依赖于F.val，我们在这两个属性之间画一条从digit.lexval指向F.val的有向边；又比如T1’.inh=T’.inh * F.val，因此T’.inh和F.val都依赖于T1’.inh，有两条有向边指向T1’.inh；如此类推画出整个依赖图。

但是我们画出依赖图的根本原因是为了用来确定一棵给定的语法分析树中各个属性实例的求值顺序的，因此我们需要计算该依赖图的拓扑序列。所谓拓扑排序是DAG（有向无环图）的所有顶点的线性序列，且要满足：
（1）每个顶点出现且只出现一次。
（2）若存在一条从顶点 A 到顶点 B 的路径，那么在序列中顶点 A 出现在顶点 B 的前面。

基于上面两个原则，可以得到写出DAG拓扑排序的一个常用方法：
（1）从 DAG 图中选择一个没有前驱（即入度为0）的顶点并输出。
（2）从图中删除该顶点和所有以它为起点的有向边。
（3）重复 1 和 2 直到当前的 DAG 图为空或当前图中不存在无前驱的顶点为止。后一种情况说明有向图中必然存在环。

语法执导的翻译方案（SDT）

SDT是在产生式右部嵌入了程序片段的CFG，这些程序片段称为语义动作。按照惯例，语义动作放在花括号内。 一个语义动作在产生式中的位置决定了这个动作的执行时间。因此可以将SDT看做是SDD的一种补充，是SDD的具体实施方案，显式地指明了语义规则的计算顺序，以便说明某些实现细节。

任何SDT都可以通过以下方法实现：首先建立一颗语法分析树，然后按照从左到右的深度优先顺序来执行这些动作，也就是说在一个前序遍历过程中执行。但通常情况下，SDT是在语法分析过程中实现的，不会真的构造一颗语法分析树。

本节主要关注如何使用SDT来实现两类重要的SDD：

  • 基本文法可以使用LR分析技术，且SDD是S属性的。
  • 基本文法可以使用LL分析技术，且SDD是L属性的。

将L-SDD转换为SDT的规则：

  • 将计算某个非终结符号A的继承属性的动作插入到产生式右部中紧靠在A的本次出现之前的位置上。

中间表示

基本块（Basic Block）

中间代码可以划分为基本块，基本块是满足下列条件的最大的连续三地址指令序列：

  • 控制流只能从基本块的第一个指令进入该块。也就是说，没有跳转到基本块中间或末尾指令的转移指令。
  • 除了基本块的最后一个指令，控制流在离开基本块之前不会跳转或者停机。

把三地址指令序列划分为基本块的方法如下：

  • 输入：一个三地址指令序列。

  • 输出：输入序列对应的基本块列表，其中每个指令恰好被分配给一个基本块。

  • 方法：首先，确定指令序列中哪些指令是首指令(leaders)，即某个基本块的第一个指令。选择首指令的规则如下：

    • 指令序列的第一个三地址指令是一个首指令。
    • 任意一个条件或无条件转移指令的目标指令是一个首指令。
    • 紧跟在一个条件或无条件转移指令之后的指令是一个首指令。

    然后，每个首指令对应的基本块包括了从它自己开始，直到下一个首指令(不含)或者指令序列结尾之间的所有指令。

流图

在PPT上控制流图（Control Flow Graph，CFG）是一个有向图G=(V, E)，节点V是基本块，边E是基本块之间的跳转关系，用来指明哪些基本块可能紧随哪些基本块之后运行。

而书本上的称之为流图(Flow Graphs)，流图是程序的一种图形化表示方式。图的结点是基本块，而图的边显示了控制流如何在基本块之间流动。流图中的循环是一个有向无环图DAG，这个区域只有一个被称为循环首结点的入口。

流图的结点是一些基本块，从基本块B到基本块C之间有一条边当且仅当基本块C的第一个指令可能紧跟在B的最后一条指令之后执行。存在这样一条边的原因有两种：

  • 有一个从B的结尾跳转到C的开头的条件或无条件跳转语句。
  • 按照原来的三地址语句序列中的顺序，C紧跟在之B后，且B的结尾不存在无条件跳转语句。

此时称B是C的前驱(predecessor) ，C是B的后继(successor)。我们通常还会增加两个分别称为“入口（entry）”和“出口（exit）”的结点。它们不和任何可执行的中间指令对应。

常用代码优化方法

优化的分类（按照涉及范围）：

  • 机器无关优化：针对中间代码。
  • 机器相关优化：针对目标代码。
  • 局部代码优化：单个基本块范围内的优化。
  • 全局代码优化：面向多个基本块的优化。

优化的分类（按照阶段）：

  • 源代码级的优化。
  • 中间代码级的优化（与机器无关）。
  • 目标代码级的优化（与机器有关）。

常用的优化方法：

  • 删除公共子表达式。
  • 删除无用代码。
  • 常量合并。
  • 代码移动。
  • 强度削弱。
  • 删除归纳变量。

在DAG上删除所有没有附加活跃变量（活跃变量是指其值可能会在以后被使用的变量）的根结点（即没有父结点的结点），重复应用这样的处理过程就可以从DAG上消除所有对应于死代码的结点。

代数恒等式的使用

第一种是化简，将计算步骤简化。 第二种是局部强度消减，把一个代价较高的运算替换为一个代价较低的运算。 第三种是常量合并，如果在编译时刻推导出一个表达式的值是常量，就可以使用该常量来替代这个表达式。

数组引用的表示

如上图，因为j可能等于i，中间的语句可能实际上改变了a[ i ]的值，因此不能认为a[ i ]是公共子表达式，然后优化为x=z。在DAG中，表示数组访问的正确方法是：

  • 从数组取值并赋给其他变量的运算（x=a[ i ]），创建一个运算符为“=[]”的结点，该结点的左右子结点分别代表数组初始值（a0）和下标（i）。该例子中的变量x是这个结点的标号之一。
  • 对数组赋值的运算（a[ j ] = y），创建一个运算符为“[]=”的结点，没有变量使用这个结点符号，这个结点有3个子结点，分别表示a0、j0和y0。
  • 该结点的创建将杀死所有已经建立的、其值依赖于a0的结点。一个被杀死的结点不能再获得任何定值变量，也就是说，它不可能成为一个公共子表达式。

数据流分析

数据流分析指的是一组用来获取有关数据如何沿着程序执行路径流动的相关信息的技术。比如实现全局公共子表达式消除的方法之一就是要求我们确定在程序的任何可能执行路径上，两个在文字上相同的表达式是否会给出相同的值。

数据流分析应用：

  • 到达定值分析(Reaching-Definition Analysis)。
  • 活跃变量分析(Live-Variable Analysis)。
  • 可用表达式分析(Available-Expression Analysis)。

在所有数据流分析应用中，都会把每个程序点和一个数据流值关联起来。

基本块上的数据流模式

到达定值

定值(Definition)：变量x的定值是(可能)将一个值赋给x的语句。

到达定值(Reaching Definition)：

  • 如果存在一条从紧跟在定值d后面的程序点到达某一程序点p的路径，并且在此路径上d没有被“杀死” (如果在此路径上有对变量x的其它定值d′，则称变量x被这个定值 d′ “杀死”了) ，则称定值d到达程序点p。
  • 直观地讲，如果某个变量x的一个定值d到达点p，在点p处使用的x的值可能就是由d最后赋予的。

用途：

  • 循环不变计算的检测：如果循环中含有赋值x=y+z ，而y和z所有可能的定值都在循环外面(包括y或z是常数的特殊情况) ，那么y+z就是循环不变计算。
  • 常量合并：如果对变量x的某次使用只有一个定值可以到达，并且该定值把一个常量赋给x，那么可以简单地把x替换为该常量。
  • 判定变量x在p点上是否未经定值就被引用。

活跃变量分析

活跃变量（Live-Variable）：
对于变量x和程序点p，如果在流图中沿着从p开始的某条路径会引用变量x在p点的值，则称变量x在点p是活跃(live)的，否则称变量x在点p不活跃(dead)。

有再次定值（且定值与本身无关），则不活跃。

用途：

  • 删除无用赋值。即如果变量x在点p的定值在基本块内所有后继点都不被引用，且x在基本块出口之后又是不活跃的，那么x在点p的定值就是无用的。
  • 为基本块分配寄存器。在一个值被计算并保存到一个寄存器中后，它很可能会在基本块中使用。如果它在基本块结尾处是死的，那就不必在结尾处保存这个值。另外，在所有寄存器都被占用时，如果我们还需要申请一个寄存器的话，那么应该考虑使用一个存放了已死亡的值的寄存器，因为这个值不需要保存到内存。

活跃变量的数据流方程 a = b + c 定义了a，引用了b, c；a = a + b 定义了a，但没有引用a。
def_B：在基本块B中定值，但是定值前在B中没有被引用的变量的集合。
use_B：在基本块B中引用，但是引用前在B中没有被定值的变量集合。

def_B和use_B的值可以直接从流图计算出来，因此在方程中作为已知量。

寄存器分配

分配思路

变量t1和t2都可以使用同一个寄存器，当且仅当在程序的任何一个执行点上变量t1和t2中至多有一个是活跃的，即如果t1和t2同时活跃，则它们不能使用同一个寄存器。

一种最简单粗暴的方法就是当新变量出现时，尝试重用一个当前不活跃变量的寄存器，如果分配失败，则把之前某个变量放回内存中，重用其寄存器。但这种方法的程序性非常差。我们希望分配算法能实现执行时间最短，分配效率高这两个目标。

基于图着色的寄存器分配算法

步骤如下：

  • 计算活跃变量分析。
  • 画出寄存器冲突图（RIG）。
  • 对RIG进行着色来分配寄存器。
  • 将无法分配的寄存器的变量溢出到内存。

计算活跃变量分析 if (e > 0)分支的活跃变量分析过程（将gen换成use，kill换成def就好理解了）：
IN[B] = GEN[B] ∪(OUT[B]-KILL[B])
OUT[B] = ∪IN[S] , S∈Succ(B)

用类似的方法可以得到整个流图的活跃变量分析结果如下：

画出寄存器冲突图 寄存器冲突图是一个无向图，每个节点是程序的一个变量，如果程序中两个变量同时活跃，添加一条边连接表示这两个变量的节点。如果两个变量间没有边连接，则它们可以使用同一个寄存器。

这样可以抽象出寄存器分配所需的所有信息，并且构建RIG后，可以进行与目标机器无关的寄存器分配算法。

对RIG进行着色来分配寄存器 给图中每个节点涂上一种颜色，使得每条边两端的节点的颜色不同，如果一个图可以被着色成k种颜色，则称这个图为k可着色的。在该算法中，颜色就是寄存器，给节点涂颜色就是给变量分配寄存器。如果RIG是k可着色的，则该程序可以使用不超过k个寄存器。 如图RIG有4种颜色，即使用4个寄存器存储了6个变量。

而计算图着色是一个非常困难的问题，目前只能使用启发式算法。对于有限数量的寄存器，可能不存在着色方案，这时候就需要对变量溢出进行讨论。

启发式算法的思路是：

  • 选择一个邻居数量小于k个的节点t。
  • 从RIG中删除t和它的所有的边。
  • 如果删除后的RIG是k可着色的，则原RIG也是k可着色的。

其步骤则是：

  • 第一部分
    • 选择一个邻居数量小于k的节点t，从RIG中删除t和它的所有的边。
    • 把t放在栈顶。
    • 重复上述步骤直到RIG中没有节点。
  • 第二部分
    • 从栈顶弹出一个节点。
    • 给该节点选择一种与其邻居都不同的颜色。
    • 重复上述步骤直到栈为空。

具体动画演示看PPT

将无法分配的寄存器的变量溢出到内存。 如果所有节点都有大于等于k个邻居，此处举例k=3，当删除a后，我们发现剩下节点的邻居都大于等于3。因此我们需要选择一个变量溢出，溢出的变量放到内存中。 假设我们选择f溢出，栈顶弹出节点着色直到遇到节点f。 此处有两种方法，一种是乐观着色法，希望给溢出节点着色时其邻居用的颜色数量正好小于3，如果是那就直接着色。另外一种是如果上述方法失败，则将该节点溢出到内存，在每次使用节点前插入load指令，在每次定义节点后插入store指令。 此时需要重新计算活跃变量分析和溢出f后的RIG。

小结 图着色方案可能需要多次溢出才能找到可行方案，而溢出哪个变量也很关键且困难，选择思路可如下：

  • 选择与其他变量冲突最多的变量。
  • 选择使用较少的变量。
  • 避免溢出循环内部的变量。
